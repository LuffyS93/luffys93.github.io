一、开发中遇到的问题

1、docker-compose遇到login to server failed:dial tcp 172.17.0.1:7897: connect: connection refused

这是我在开发时遇到的第一个问题，除了自己的服务器，在堡垒机上搭建也是遇到了这个问题，我搜了很多资料，问题的原因要么是你这个端口或者ip被占用了，要么就是服务没搭起来所以才会导致connection-refused

这个问题在CTFd的体现就是如下画面：（这里用到了glzjin师傅开发的CTFd-whale插件）

我在向我的学长问这个问题时，他开始判断的是docker的子网被占了，之前我也查了资料，问了gpt，确实有这个可能性，但是我服务器是新买的，应该不存在子网被占用的问题。所以在我用ip addr与ip a查看网络状态后发现这子网都是它们自己在用，不存在被其他进程占用子网的可能性。

下图就是ip a之后的信息

后面我转到堡垒机上弄平台发现也是这个问题，问了一下之后才知道哥们的服务就没搭起来，一直是failed（给人家整无语了）

后面为了方便，我就改成默认的名字，结果也还是不行，后面才知道要先删除原有的服务，然后再重新加载上去（大概就是先disable再enable）

出现这个问题多半是在重构镜像后进行docker-compose up，我猜是出现了类似于旧容器和重构镜像冲突的问题，把旧的容器删了重新用新镜像弄一个就可以了

3、 在docker-compose的时候总是会遇到类似于login failed情况（反正就是token对不上）

这种时候往往不是/etc/frp/frps.toml的问题，需要注意看看CTFd/conf/compose-frp/frpc.ini或着CTFd/conf/frp/frpc.ini的token有没有对上，后者的token记得不要加引号（因为解析的时候会把引号当做一个字符来处理，虽然不知道原理是啥样的）

还有另外的情况没遇到，有可能是0.52版本之后对于json、toml、ini等配置文件的更新带来的影响（.ini文件是要被废除掉的，这里就看官方文档怎么说了）

二、服务器测试

1、压力测试

这里使用Apache Benchmark来进行短期大流量的压力测试，内容为：模拟100个用户发送总共10000条请求

不过啊，这个结果已经让用户觉得服务器很卡，性能不好。这里分析一下

从上至下依次为：

  • 用户并发数（Concurrency Level）
  • 测试时长（Time taken for tests）
  • 完成/失败 请求数（Complete/Failed requests）
  • 传输总量（Total transferred）
  • HTML传输总量（HTML transferred）
  • 每秒请求数（Requests per second）
  • 平均请求时间（Time per request）
  • 每个用户的请求时间（Time per request，across all concurrent requests）
  • 传输速率（Transfer rate）

虽然传输过程中没有出错，但是平均请求时间都2秒多了，肉眼可见的卡

这个则是连接时间的分布，以毫秒为单位（中间的[+/-sd]是标准差）

连接方面的话，服务器能够做出快速响应，但是在等待和加载的过程中波动巨大，我猜最小值有几十到一百的原因是：刚开始还没多少用户，所以服务器相对来说是处于一个比较空闲的状态，能够应付的过来，后面最大值有两千多的原因估计就是应付不过来了，只能排队等候

这里就能体现出我上面说的了，请求数太多导致有点应付不过来（当然也有可能是用户太多，不过平均下来一个用户在这段时间发送100条请求也感觉有点够多了）

下面这个就更夸张了，模拟700个用户发送10000条请求

当时我在测试的时候打开页面就能感受到了，刷新和请求题目至少都需要20秒:sob:

我们的要求是刷新页面不能太卡，而且在请求查看某个题目或者其他东西的时候要能及时出现，看看能不能争取弄到1秒以下吧

后面还用了一个东西，但是忘了叫什么了，在测试的时候发现服务器是单核运作的，然而服务器是8核，在询问战神之后得知是docker-compose的参数没改，把workers改高一点就能多线程了

CodeInject

正常输入是要将你输入的数据转换成Object类型并以var_dump形式输出，所以肯定不会执行你输入的命令

不过呢这似乎存在一个闭合问题呢，试着传入1);//

那么原来的代码就可以等效为eval(“var_dump((object)1);//);”)

注意了，这里的//是被当做字符串处理了，所以它不会过滤外部的语句，由于eval()是把传入的参数当做php语句进行处理，相当于在<?php xxxx ?>中把xxxx替换为传入的参数。所以//过滤的实际上是var_dump((object)1);//);中的);

成功闭合后就可以开始输入命令了

学习web安全的那些二三事-反弹Shell篇

反弹Shell也是一个需要掌握的技巧（实际上我也没掌握到多少，还搞得好混乱，只有找题来看看）

在有反弹shell之前，技术人员连接目标机器采用的是老早的传统连接方法，例如有Telnet、ssh、远程桌面协议等等，还有其他的第三方远程管理工具如VNC与一些批处理脚本。这些连接方式可以称为主动连接。

不过，有的目标机器会有一些限制：

  1. 目标机器对连接进行了限制，即允许出站连接，但限制了很多入站连接，这作用与内网无异；
  2. 目标机器的某端口被占用，由于其本身的防火墙策略会导致ssh连接被阻断；
  3. 目标机器的ip是动态的，或者处于内网状态下（即目标机器处于局域网之中，ip地址由NAT管理）；

反弹shell就是为了绕过上述条件而产生的，原理就是让目标机器主动连接外部设备。

反弹shell的方法有很多种，包括但不限于python脚本反弹，netcat反弹，bash反弹等等。
